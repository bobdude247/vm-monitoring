# Incident Report: VS Code Lockup + Return to Login Screen (2026-04-05)

## Summary

This report captures findings from a post-incident diagnostic run after a VS Code lockup that was followed by an unexpected return to the Kali VM login screen.

- **Incident date/time (local):** 2026-04-05 ~23:29–23:36 EDT
- **Host/guest:** Kali Linux VM in VirtualBox
- **Observed behavior:** VS Code became unresponsive; desktop session returned to login.
- **Most likely pattern:** **X/desktop session reset under graphics-path instability + memory/swap pressure**, not a kernel panic.

## Diagnostic Artifact

- Recovery report generated by:

```bash
sudo bash monitoring/post-recovery-report.sh
```

- Saved as:

```text
/var/log/vm-recovery-reports/recovery-2026-04-05-233604.txt
```

## Evidence Reviewed

### 1) Kernel / reboot / panic signals

- No direct kernel panic signature found.
- No OOM-killer signature found in the previous-boot filtered section.
- No watchdog reset signature found.
- The behavior aligns with session teardown/restart more than full-kernel failure.

### 2) Graphics / virtualization warnings (high signal)

Previous-boot journal contains repeated warnings such as:

- `vmwgfx ... unsupported hypervisor`
- `This configuration is likely broken`

These warnings are consistent with prior incidents where the VM display/session path was implicated.

### 3) Resource pressure around incident window (high signal)

The vmstat/top snapshots near incident time show:

- Several heavy `code` processes with high CPU and large RSS.
- Elevated swap usage (`swpd` in hundreds of MB).
- Active reclaim/swap activity (`si/so` nonzero).
- `kswapd0` active during pressure windows.
- Low free memory periods.

This indicates memory contention and swap thrash during active workload.

## Interpretation

The strongest explanation for this event is:

1. **High interactive load from multiple VS Code processes** caused significant memory pressure.
2. Under pressure, desktop responsiveness degraded.
3. Existing **VirtualBox graphics/X stack instability** increased likelihood of a session collapse/reset.
4. User experienced return to login screen (session reset), without clear evidence of full VM kernel panic.

## Recommended Actions (Priority Order)

1. **Stabilize VirtualBox graphics path**
   - Test with 3D acceleration **disabled** first.
   - Keep VMSVGA unless performing controlled A/B testing.
   - Maximize VM video memory within safe profile limits.
   - Ensure Guest Additions and host VirtualBox versions are aligned.

2. **Increase VM memory headroom**
   - Raise VM RAM allocation if host capacity allows.
   - Keep at least 2 vCPUs, avoid host overcommit.

3. **Reduce VS Code footprint in VM**
   - Fewer concurrent VS Code windows/workspaces.
   - Disable non-essential extensions for this VM profile.
   - Exclude very large directories from indexing/search/watch.

4. **Lower guest desktop overhead**
   - Close non-essential GUI apps/tabs during coding.
   - Reduce background/tray services where possible.

5. **Continue forensic capture on repeats**
   - Run post-incident report immediately after each recurrence.
   - Compare with previous incident notes for trend consistency.

## Notes

- `sudo` is required for full report capture because the script reads privileged system logs under `/var/log` and previous-boot journal content.
- The concurrent `zsh` history corruption warning is likely from unclean shutdown/session interruption and should be remediated separately; it is not the primary crash signature.
